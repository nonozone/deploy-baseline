# 通用部署 SOP

## 1. 文档目的

这份 SOP 用于统一不同项目的部署规范写法和执行流程。

以后每个项目都应基于这份 SOP 生成自己的部署文档，而不是各自自由发挥。

这份 SOP 只定义通用流程，不绑定具体业务项目。

## 2. 适用对象

适用于以下角色：

- 项目负责人
- 开发人员
- 运维人员
- 需要接手项目部署的后来维护者

## 3. 项目部署文档必须包含的章节

每个项目的 `deploy/README.md` 或同级部署文档，至少应包含下面内容：

1. 部署目标与适用范围
2. 服务器或运行环境前置条件
3. 本地运行模式说明
4. 服务发布拆分说明
5. 标准目录约定
6. 环境文件说明
7. 部署前检查
8. 标准发布流程
9. 启动契约与发布完成判定
10. 日志查看与健康检查
11. 回滚流程
12. 常见故障排查
13. 项目特有补充项

## 4. 标准目录约定

建议项目部署目录至少包含：

```text
deploy/
├── README.md
├── env/
│   └── app.env.example
└── scripts/
    ├── preflight.sh
    ├── deploy.sh
    └── rollback.sh
```

要求：

- `deploy/README.md` 为项目部署文档入口
- `deploy/env/` 只放样例文件，不放真实密钥
- `deploy/scripts/` 放可执行脚本

如果项目还需要状态型服务自检，也可以增加：

- `deploy/scripts/check-storage.sh`
- `scripts/check_db_persistence.sh`

## 5. 环境文件约定

### 5.1 基本要求

- 本地开发变量与部署变量必须分离
- 部署文档必须说明真实生产变量存放方式
- 样例文件必须说明哪些值需要替换

### 5.2 最低要求

项目部署文档中必须写清：

- 应用服务变量来自哪里
- 数据库变量来自哪里
- 敏感变量如何注入
- 是否使用 `--env-file`
- 发布端口由哪些变量控制

## 6. 本地运行模式说明

项目部署文档必须写清当前项目采用哪一种本地运行模式：

- 全量 Docker
- 混合开发

如果是混合开发，还必须写清：

- 哪些服务走 Docker
- 哪些服务本地运行
- `make dev` 会启动哪些内容
- 开发人员还需要额外执行哪些命令

## 7. 服务发布拆分说明

项目部署文档必须写清不同服务是否共用同一条发布链路。

至少要说明：

- 哪些服务归 `make deploy` 管理
- 哪些服务是独立发布
- 静态资源是否独立发布
- 数据库是否在同一 Compose 中维护

这一步的目的是防止“顶层命令统一，但真实发布路径没人说得清”。

## 8. 部署前检查

每个项目都必须提供一个 `make deploy-check` 或等价检查入口。

项目应把 `deploy/env/app.env.example` 作为唯一 env 示例来源，并提供非破坏性的 env sync 入口，例如：

```bash
make prod-env-sync
```

要求：

- 只追加真实 `.env` 或 `app.prod.env` 中缺失的 key
- 不覆盖已有值
- 在部署检查中能明确提示 example 与真实 env 的结构漂移

部署前至少检查以下内容：

1. Docker 可用
2. Docker Compose 可用
3. 生产 Compose 文件存在
4. 部署环境文件存在
5. 关键变量已配置
6. 挂载目录或命名卷策略明确
7. 目标服务名称与脚本一致

建议把这些检查收敛到 `deploy/scripts/preflight.sh`。

如果项目包含数据库或存储等状态型依赖，建议额外补充：

- 持久化目录检查
- 命名卷检查
- 运行中挂载检查

建议同时补充最低验证面：

- `bash -n` 检查新增或修改过的 shell 脚本
- `docker compose config` 检查 Compose 展开结果
- `make help` 检查命令面
- 如果项目已有 `build/test/typecheck`，则执行现有健康检查

## 9. 标准发布流程

推荐所有项目都按以下顺序发布：

1. 执行部署前检查
2. 拉取或构建最新镜像
3. 使用生产 Compose 文件启动或更新服务
4. 等待关键服务进入健康状态
5. 执行健康检查
6. 查看日志
7. 记录本次发布版本或镜像标签

推荐命令入口：

```bash
make deploy-check
make deploy
make logs
```

如果项目需要额外步骤，例如静态资源上传、数据库迁移、缓存预热，也必须在部署文档中明确写出顺序。

## 10. 启动契约与发布完成判定

每个项目部署文档中必须写清：

- 服务启动前需要等待什么依赖
- 是否自动执行数据库迁移
- 启动契约变量有哪些
- 什么条件下才算部署完成

建议覆盖：

- `WAIT_FOR_DB`
- `RUN_DB_MIGRATIONS`
- `DB_WAIT_TIMEOUT`
- `APP_HEALTHCHECK_PATH`
- `APP_HEALTHCHECK_TIMEOUT`
- 健康检查接口或探针

推荐默认约定：

- 健康检查路径为 `/health`
- 成功标准为返回 `200`
- 发布完成判定至少包含“容器 healthy”或等价探针成功

## 11. 日志与健康检查

每个项目部署文档中必须说明：

- 应该查看哪个服务日志
- 使用什么命令查看
- 健康检查接口或探针是什么
- 发布后如何确认服务可用

最低要求：

- 提供日志查看命令
- 提供健康检查方式
- 提供发布完成判定方式
- 提供失败后的第一排查入口

## 12. 回滚流程

每个项目必须明确回滚边界，不能只写“有问题就回滚”。

至少应写清：

- 回滚单位是什么
  - 镜像标签
  - Git 版本
  - 制品版本
- 回滚前提是什么
- 回滚命令是什么
- 回滚后如何验证

推荐约定：

- 容器镜像优先使用 Git Commit SHA 或语义化版本号作为 tag
- 不建议使用 `latest` 作为生产发布或回滚依据
- 回滚时应明确写出目标版本，而不是依赖隐式默认值

推荐统一入口：

```bash
make rollback
```

如果模板里的 `rollback.sh` 只是占位实现，那么目标项目在接入时必须补齐真实回滚逻辑。

## 13. 常见故障排查

项目部署文档中建议至少覆盖以下问题：

- 环境文件缺失
- 端口冲突
- 容器启动失败
- 健康检查失败
- 数据卷挂载错误
- 数据库未就绪
- 镜像标签配置错误

## 14. 项目生成部署规范时必须补充的项目特有信息

这份通用 SOP 不能替代项目自己的部署说明。每个项目必须额外补充：

- 项目名称
- 服务名称
- 镜像名称
- 实际发布端口
- 实际健康检查地址
- 数据库与存储依赖
- 真实部署命令
- 回滚命令或回滚策略
- 推荐的版本命名方式
- 项目特有的前置准备步骤

如果项目包含状态型服务或系统级配置，还建议额外补充：

- 版本陷阱与升级注意事项
- 持久化目录或命名卷变更风险
- 反向代理、systemd、CI/CD 等系统级配置是否采用 merge 还是整体替换

## 15. 推荐的项目部署文档骨架

项目部署文档建议采用下面结构：

```markdown
# 项目部署规范

## 1. 适用范围
## 2. 前置条件
## 3. 本地运行模式说明
## 4. 服务发布拆分说明
## 5. 目录与文件约定
## 6. 环境变量说明
## 7. 部署前检查
## 8. 标准发布流程
## 9. 启动契约与发布完成判定
## 10. 日志与健康检查
## 11. 回滚流程
## 12. 常见故障排查
## 13. 项目特有说明
```

## 16. 执行要求

以后其他项目生成部署规范时，应同时满足下面三点：

1. 文档结构遵守本 SOP
2. 顶层命令遵守通用基线规范
3. 模板脚本与项目实际部署流程保持一致
