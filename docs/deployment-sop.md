# 通用部署 SOP

## 1. 文档目的

这份 SOP 用于统一不同项目的部署规范写法和执行流程。

以后每个项目都应基于这份 SOP 生成自己的部署文档，而不是各自自由发挥。

这份 SOP 只定义通用流程，不绑定具体业务项目。

## 2. 适用对象

适用于以下角色：

- 项目负责人
- 开发人员
- 运维人员
- 需要接手项目部署的后来维护者

## 3. 项目部署文档必须包含的章节

每个项目的 `deploy/README.md` 或同级部署文档，至少应包含下面内容：

1. 部署目标与适用范围
2. 服务器或运行环境前置条件
3. 标准目录约定
4. 环境文件说明
5. 部署前检查
6. 标准发布流程
7. 日志查看与健康检查
8. 回滚流程
9. 常见故障排查
10. 项目特有补充项

## 4. 标准目录约定

建议项目部署目录至少包含：

```text
deploy/
├── README.md
├── env/
│   ├── app.env.example
│   └── app.prod.env.example
└── scripts/
    ├── preflight.sh
    ├── deploy.sh
    └── rollback.sh
```

要求：

- `deploy/README.md` 为项目部署文档入口
- `deploy/env/` 只放样例文件，不放真实密钥
- `deploy/scripts/` 放可执行脚本

## 5. 环境文件约定

### 5.1 基本要求

- 本地开发变量与部署变量必须分离
- 部署文档必须说明真实生产变量存放方式
- 样例文件必须说明哪些值需要替换

### 5.2 最低要求

项目部署文档中必须写清：

- 应用服务变量来自哪里
- 数据库变量来自哪里
- 敏感变量如何注入
- 是否使用 `--env-file`
- 发布端口由哪些变量控制

## 6. 部署前检查

每个项目都必须提供一个 `make deploy-check` 或等价检查入口。

部署前至少检查以下内容：

1. Docker 可用
2. Docker Compose 可用
3. 生产 Compose 文件存在
4. 部署环境文件存在
5. 关键变量已配置
6. 挂载目录或命名卷策略明确
7. 目标服务名称与脚本一致

建议把这些检查收敛到 `deploy/scripts/preflight.sh`。

## 7. 标准发布流程

推荐所有项目都按以下顺序发布：

1. 执行部署前检查
2. 拉取或构建最新镜像
3. 使用生产 Compose 文件启动或更新服务
4. 查看日志
5. 执行健康检查
6. 记录本次发布版本或镜像标签

推荐命令入口：

```bash
make deploy-check
make deploy
make logs
```

如果项目需要额外步骤，例如静态资源上传、数据库迁移、缓存预热，也必须在部署文档中明确写出顺序。

## 8. 日志与健康检查

每个项目部署文档中必须说明：

- 应该查看哪个服务日志
- 使用什么命令查看
- 健康检查接口或探针是什么
- 发布后如何确认服务可用

最低要求：

- 提供日志查看命令
- 提供健康检查方式
- 提供失败后的第一排查入口

## 9. 回滚流程

每个项目必须明确回滚边界，不能只写“有问题就回滚”。

至少应写清：

- 回滚单位是什么
  - 镜像标签
  - Git 版本
  - 制品版本
- 回滚前提是什么
- 回滚命令是什么
- 回滚后如何验证

推荐统一入口：

```bash
make rollback
```

如果模板里的 `rollback.sh` 只是占位实现，那么目标项目在接入时必须补齐真实回滚逻辑。

## 10. 常见故障排查

项目部署文档中建议至少覆盖以下问题：

- 环境文件缺失
- 端口冲突
- 容器启动失败
- 健康检查失败
- 数据卷挂载错误
- 数据库未就绪
- 镜像标签配置错误

## 11. 项目生成部署规范时必须补充的项目特有信息

这份通用 SOP 不能替代项目自己的部署说明。每个项目必须额外补充：

- 项目名称
- 服务名称
- 镜像名称
- 实际发布端口
- 实际健康检查地址
- 数据库与存储依赖
- 真实部署命令
- 回滚命令或回滚策略
- 项目特有的前置准备步骤

## 12. 推荐的项目部署文档骨架

项目部署文档建议采用下面结构：

```markdown
# 项目部署规范

## 1. 适用范围
## 2. 前置条件
## 3. 目录与文件约定
## 4. 环境变量说明
## 5. 部署前检查
## 6. 标准发布流程
## 7. 日志与健康检查
## 8. 回滚流程
## 9. 常见故障排查
## 10. 项目特有说明
```

## 13. 执行要求

以后其他项目生成部署规范时，应同时满足下面三点：

1. 文档结构遵守本 SOP
2. 顶层命令遵守通用基线规范
3. 模板脚本与项目实际部署流程保持一致
