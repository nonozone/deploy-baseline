# `deploy-baseline-kit` 设计说明

## 1. 目标

`deploy-baseline-kit` 是一个单入口主 skill，用于识别任意项目目录，并基于当前仓库定义的部署基线，对项目进行生成或收敛改造。

它的目标不是只做检查，而是统一接管以下工作：

- 判断项目根目录
- 识别当前项目部署现状
- 生成改造方案
- 在开发者一次确认后执行改造
- 输出改造结果、偏离项和后续说明

## 2. 设计原则

### 2.1 单入口

对外只暴露一个主 skill，不要求开发者逐个调用子 skill 或手工串流程。

### 2.2 先分析，后改造

主 skill 触发后默认先输出识别结果和改造方案，不直接修改文件。

只有在开发者确认后，才进入改造执行阶段。

### 2.3 只保留一个确认点

无论项目简单还是复杂，整个流程都只保留一次人工确认，避免反复打断。

### 2.4 自动判断优先

项目根目录、运行模式、改造路径都应优先自动推断。只有在无法可靠判断时，才向开发者补充确认。

### 2.5 强制收敛可用

对于已有项目，主 skill 可以支持强制收敛到基线结构。

如果检测到现有部署配置较重，主 skill 需要在唯一确认点中向开发者明确说明风险，并允许开发者选择：

- 保守改造
- 强制收敛

## 3. 适用范围

`deploy-baseline-kit` 适用于以下两类输入对象：

- 近似空目录或尚未建立容器部署体系的项目目录
- 已有项目目录，包括已经存在部分或完整部署配置的项目

它不要求目标目录预先具备 Compose、Makefile 或 deploy 结构。

## 4. 主流程

主 skill 的固定流程如下：

1. 根目录识别
2. 项目现状扫描
3. 与部署基线的差距分析
4. 生成改造方案并请求一次确认
5. 在确认后执行改造
6. 输出变更摘要、基线符合项和项目特例

## 5. 根目录识别规则

开发者可能在项目根目录运行 skill，也可能在任意子目录运行 skill。

因此第一步必须先识别目标项目根目录，而不是直接假设当前工作目录就是项目根。

识别规则建议为：

- 向上查找最像项目根的目录
- 优先识别工程锚点，例如 `.git/`、`package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`Dockerfile`、`Makefile`、`README.md`
- 如果当前目录缺少关键锚点、而上层目录具备更完整工程结构，则以上层目录作为候选根
- 如果存在多个根候选，例如 monorepo、多应用仓库或前后端分层仓库，则在确认点中明确提示识别结果与建议作用范围

## 6. 项目现状分类

主 skill 至少需要把目标项目识别为以下三类之一：

### 6.1 空目录或近似空目录

特点：

- 几乎没有工程锚点
- 没有 Compose
- 没有 `Makefile`
- 没有 `deploy/`

处理方式：

- 直接进入基线骨架生成方案

### 6.2 轻量已有项目

特点：

- 已有应用代码或基础工程文件
- 可能缺少完整部署入口
- 可能只有部分脚本、单个 Compose 文件或简单 `Makefile`

处理方式：

- 输出缺失项和收敛方案
- 确认后补齐并统一结构

### 6.3 重度已有部署项目

特点：

- 已有较完整的 `Makefile`
- 已有多个 Compose 文件或复杂部署脚本
- 已有 `deploy/` 结构或项目特有发布链路

处理方式：

- 输出识别结果、冲突点、预期重写范围
- 在唯一确认点中让开发者选择保守改造或强制收敛

## 7. 运行模式判断

主 skill 默认自动判断项目更适合以下哪种模式：

- 全量 Docker
- 混合开发

判断建议：

- 如果核心服务明显都通过 Compose 管理，优先判为全量 Docker
- 如果数据库、缓存等依赖走容器，但应用服务已有本地开发启动方式，优先判为混合开发
- 如果证据不足，则在唯一确认点中说明无法可靠自动判断，并请开发者确认模式

## 8. 数据库策略

主 skill 不能把 PostgreSQL 视为所有项目的默认事实。

它需要先判断目标项目属于哪一类数据存储场景：

- 项目自带数据库容器
- 项目使用外部托管数据库
- 项目当前没有明确数据库依赖

如果项目自带数据库容器，还应进一步识别数据库类型，例如：

- PostgreSQL
- MySQL
- MariaDB
- MongoDB
- 其他项目特有数据库

识别依据可以包括：

- 现有 Compose 服务定义
- 环境变量命名
- 应用配置文件
- 依赖包与 ORM 配置
- README、部署文档或脚本中的数据库线索

如果项目当前使用的是外部托管数据库，而不是本地自带数据库容器，则主 skill 不应强行为项目生成本地数据库容器，只需要为环境变量、部署文档和启动契约补齐对应说明。

如果是全新项目或近似空目录，且无法从项目现状中识别数据库类型，则应在唯一确认点中明确要求开发者确认数据库类型，而不是默认假定所有项目都使用 PostgreSQL，也不是默认不生成数据库容器。

## 9. 确认点设计

整个流程只保留一个确认点。

### 9.1 轻量项目

输出内容包括：

- 项目根识别结果
- 当前已有资产
- 与基线的差距
- 建议执行的改造方案

然后请求一次确认，再执行改造。

### 9.2 重度已有部署项目

输出内容包括：

- 项目根识别结果
- 当前已有部署资产
- 与基线的主要冲突点
- 预计会重写、迁移、保留的内容摘要

然后在同一次确认中让开发者选择：

- 保守改造
- 强制收敛

确认后不再逐文件追问。

## 10. 改造执行规则

确认后，主 skill 应按部署基线统一改造项目结构。

目标结构包括但不限于：

- `Makefile`
- `.env.example`
- `docker-compose.yml`
- `docker-compose.dev.yml`
- `docker-compose.prod.yml`
- `scripts/`
- `deploy/`
- `deploy/env/`
- `deploy/scripts/`
- `deploy/README.md`

执行规则如下：

- 基线目录结构是目标结果，不是可选建议
- 可以重写或收敛现有部署入口，但不能静默丢弃明显存在的项目特有逻辑
- 对项目特有逻辑，应优先迁移、兼容、备份或在文档中标注
- 无法可靠自动判断的事项，只能在唯一确认点中询问，确认后不再展开逐项确认

### 10.1 数据库变体规则

数据库基线应按数据库类型区分，而不是只围绕 PostgreSQL 设计。

规则如下：

- 如果项目已经明确使用某种数据库，主 skill 应优先沿用该数据库类型生成或改造基线
- 如果项目使用外部托管数据库，则不强制生成数据库容器，但要补齐变量、文档和部署说明
- 如果项目无法识别数据库类型，则在唯一确认点中要求开发者确认，而不是静默套用 PostgreSQL 模板，也不是默认跳过数据库部分

### 10.2 PostgreSQL 基线版本

如果主 skill 识别出项目采用 PostgreSQL，并且目标方案需要 PostgreSQL 容器基线，则默认基线版本应采用：

- `postgres:18-alpine`

规则如下：

- 对于全新项目，只有在确认目标数据库类型为 PostgreSQL 时，才默认使用 `postgres:18-alpine`
- 对于已有项目，不能机械地把现有 PostgreSQL 版本直接替换为 `postgres:18-alpine`
- 主 skill 应结合项目已有 Compose、环境变量、数据目录、扩展依赖和部署约束，判断应采用哪一种处理方式

对于已有 PostgreSQL 项目，允许的处理结果包括：

- 保持现有版本，并在改造方案或结果说明中标注原因
- 提示项目可以升级到 `postgres:18-alpine`，但不自动执行版本跃迁
- 在具备足够依据且开发者确认改造方案后，执行相应版本调整

也就是说，`postgres:18-alpine` 是 PostgreSQL 变体的新基线默认值，但不是所有项目的通用默认数据库，也不是对已有项目一刀切强升的硬规则。

## 11. 输出契约

主 skill 的输出必须固定包含两类结果。

### 11.1 改造前

- 项目根识别结果
- 当前状态摘要
- 差距清单
- 建议改造路径
- 数据库类型识别结果或无法识别说明

### 11.2 改造后

- 本次修改摘要
- 已对齐的基线项
- 仍保留的项目特例
- 偏离基线的项及原因

## 12. 资源组织建议

建议 skill 采用以下目录结构：

```text
deploy-baseline-kit/
├── SKILL.md
├── references/
│   ├── root-detection.md
│   ├── project-analysis.md
│   ├── mode-detection.md
│   ├── database-variants.md
│   ├── transformation-rules.md
│   └── document-generation.md
├── assets/
│   └── template/
└── scripts/
    └── detect-root.sh
```

职责建议如下：

- `SKILL.md`：放触发条件、主流程、确认点和执行规则
- `references/`：放判断逻辑、数据库变体规则、转换规则和文档生成规则
- `assets/template/`：复用本仓库模板，作为默认骨架来源
- `scripts/detect-root.sh`：提升根目录识别的稳定性

## 13. 命名

正式名称采用：

- `deploy-baseline-kit`

该命名强调它是围绕部署基线标准提供的落地工具包，而不是单纯的规范文档映射。
