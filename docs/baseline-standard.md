# 容器部署与 Make 通用基线规范

## 1. 目标

这份规范用于统一不同项目的容器部署方式、环境变量管理方式和 `Makefile` 操作入口。

目标不是约束项目技术栈，而是统一工程接口，确保团队在切换项目时：

- 目录结构容易识别
- 常用命令保持一致
- 部署文档格式一致
- 新项目能快速复制落地

## 2. 适用范围

这套规范适用于以下类型项目：

- 使用 Docker Compose 管理本地开发环境或生产部署
- 使用 `make` 统一开发、构建、测试或部署入口
- 存在一个或多个应用服务，并可能依赖数据库、缓存、前端、Worker 或网关

不限制后端语言和前端框架，但要求在工程接口层遵守同一套命名规则。

## 3. 核心原则

### 3.1 命令接口统一

不同项目必须尽量收敛为相同的顶层命令。最少应保留：

- `make dev`
- `make build`
- `make test`
- `make deploy`
- `make rollback`
- `make logs`

项目可以保留扩展命令，例如：

- `make db-reset`
- `make migrate`
- `make seed`
- `make build-admin`
- `make build-web`

但扩展命令不能替代统一顶层命令。

这些顶层命令不只是“名字相同”，还应尽量保持语义一致：

- `make dev`：项目默认开发入口
- `make build`：项目级默认构建入口
- `make test`：项目级默认测试入口
- `make deploy`：标准部署入口
- `make rollback`：标准回滚入口
- `make logs`：默认日志查看入口

如果项目让这些命令承担完全不同的含义，就会破坏跨项目切换时的可预期性。

### 3.2 容器负责运行环境统一

容器和 Compose 文件负责描述：

- 服务如何启动
- 服务之间如何依赖
- 端口如何暴露
- 数据如何持久化
- 健康检查如何执行

不要把运行环境约定分散写进 README 或随手脚本中。

### 3.3 Make 负责操作入口统一

`Makefile` 的职责是暴露操作入口，不是承载复杂实现逻辑。

规则如下：

- `Makefile` 只做命令收口
- 复杂逻辑下沉到 `scripts/` 或 `deploy/scripts/`
- 每个 target 必须带中文说明
- 必须提供 `make help`

### 3.4 文档与模板分离

规范文档负责定义标准，模板负责提供默认实现。

不要把“项目说明”“规范说明”“模板使用说明”混写在一个 README 中。

### 3.5 渐进迁移

旧项目不要求一次性完全重构。

建议迁移顺序：

1. 先统一命令名
2. 再统一 Compose 分层
3. 再统一环境变量来源
4. 最后统一部署文档与回滚流程

### 3.6 允许多种运行与发布模式

统一的是工程接口，不是要求所有项目必须采用同一种运行形态。

在符合统一命令和统一文档结构的前提下，允许项目采用：

- 全量 Docker 开发模式
- 混合开发模式
- 分离式发布模式

但项目必须把自己的模式写清楚，不能依赖口头约定。

## 4. 标准目录结构

建议项目按以下结构组织：

```text
project-root/
├── Makefile
├── .env.example
├── docker-compose.yml
├── docker-compose.dev.yml
├── docker-compose.prod.yml
├── deploy/
│   ├── README.md
│   ├── env/
│   │   ├── app.env.example
│   │   └── app.prod.env.example
│   └── scripts/
│       ├── preflight.sh
│       ├── deploy.sh
│       └── rollback.sh
├── scripts/
│   ├── setup.sh
│   ├── dev.sh
│   ├── build.sh
│   └── test.sh
└── docs/
    └── engineering/
        └── deployment-baseline.md
```

如果项目使用混合开发模式，也可以按需增加本地运行辅助脚本，例如：

- `scripts/dev-local.sh`
- `scripts/dev-admin.sh`
- `scripts/dev-storefront.sh`

## 5. 文件职责

### 5.1 `Makefile`

负责统一入口：

- 开发
- 构建
- 测试
- 部署
- 回滚
- 日志

不建议把长命令、复杂条件判断、业务流程直接塞进 `Makefile`。

### 5.2 `docker-compose.yml`

负责放跨环境共享内容，通常包括：

- 数据库
- 缓存
- 公共 volume
- 公共 network
- 可复用的基础 service 定义

### 5.3 `docker-compose.dev.yml`

负责放开发态差异，例如：

- 热更新
- 本地源码挂载
- 调试开关
- 本地端口暴露

### 5.4 `docker-compose.prod.yml`

负责放生产态差异，例如：

- 发布端口
- 重启策略
- 持久化目录
- 生产环境变量
- 健康检查

### 5.5 `.env.example`

负责本地开发变量示例，不放真实敏感信息。

### 5.6 `deploy/env/*.example`

负责部署阶段变量示例，强调部署变量与本地开发变量分层管理。

### 5.7 `scripts/`

负责本地开发与构建辅助脚本。

### 5.8 `deploy/scripts/`

负责部署前检查、部署、回滚等线上操作脚本。

### 5.9 `deploy/README.md`

负责项目自己的部署说明文档。文档结构应遵守通用部署 SOP。

### 5.10 `docs/engineering/deployment-baseline.md`

负责记录项目对通用基线的具体落地方式，尤其是：

- 本地运行模式
- 服务发布拆分方式
- 沿用的标准项
- 偏离标准的地方及原因

## 6. Compose 分层规则

推荐采用三层结构：

### 6.1 基础层

`docker-compose.yml`

作用：

- 定义跨环境共享服务
- 定义命名卷
- 定义默认网络

### 6.2 开发层

`docker-compose.dev.yml`

作用：

- 叠加开发模式特有服务
- 叠加开发端口与源码挂载
- 叠加本地调试参数

### 6.3 生产层

`docker-compose.prod.yml`

作用：

- 叠加生产发布配置
- 叠加持久化策略
- 叠加健康检查与重启策略

### 6.4 允许的本地运行模式

为了兼顾不同项目规模，基线允许至少两种本地运行模式：

#### 模式 A：全量 Docker

所有核心服务都通过 Compose 启动。

适合：

- 单服务项目
- 后端与前端都已经容器化
- 希望开发环境最一致的项目

#### 模式 B：混合开发

一部分服务通过 Docker 启动，另一部分服务直接在本地开发机启动。

常见场景：

- 后端和数据库走 Docker
- 前端本地热更新运行
- 数据库走 Docker，Worker 或前端本地运行

要求：

- 顶层入口仍然使用 `make dev`
- 项目必须在文档中明确哪些服务走 Docker，哪些服务走本地
- 项目必须说明 `make dev` 是否会同时拉起本地进程

### 6.5 允许的服务发布拆分方式

基线允许一个项目的不同服务使用不同发布路径。

例如：

- `api` 通过 Docker Compose 发布
- `admin` 通过静态站点服务发布
- `storefront` 通过 CDN 或静态资源服务发布

要求：

- 项目必须明确哪些服务归 `make deploy` 管理
- 未纳入 `make deploy` 的服务，也必须在部署文档中写清真实发布方式
- 顶层命令统一，不代表所有服务必须使用同一技术路径

### 6.6 生产健康检查与启动完成判定

生产 Compose 配置不应只有启动命令，还应显式定义：

- 服务健康检查
- 服务何时算 ready
- 数据库依赖就绪条件

建议：

- `db` 服务配置健康检查
- 应用服务依赖 `service_healthy`
- 应用服务提供自身健康检查
- 部署文档中写清发布后如何验证服务可用

### 6.7 状态型服务自检

对于数据库、存储、消息队列等状态型服务，建议提供额外自检脚本。

最低应关注：

- 数据是否落在预期命名卷或挂载目录
- 核心路径是否与配置一致
- 运行中容器是否真的使用了预期配置

这类检查可以放在：

- `scripts/*.sh`
- `deploy/scripts/*.sh`

## 7. 环境变量规则

### 7.1 分层原则

环境变量至少分为两层：

1. 本地开发变量
2. 部署运行变量

需要时可以继续拆分：

- 构建变量
- 预发布变量
- 生产敏感变量

### 7.2 约束规则

- 本地开发变量放在 `.env.example`
- 部署变量样例放在 `deploy/env/*.example`
- 敏感信息不能提交进仓库
- Compose 文件只引用变量，不写死密钥
- 能提供默认值的变量，应明确写出默认值来源

### 7.3 命名建议

- 端口统一使用 `*_PORT`
- 监听地址或发布地址统一使用 `*_HOST` 或 `*_URL`
- 生产发布 IP 和端口单独定义，例如 `APP_PUBLISH_IP`、`APP_PUBLISH_PORT`
- 布尔值显式写为 `true` 或 `false`

### 7.4 启动契约变量

对于容器化应用，建议显式定义一组启动契约变量，而不是把启动前置逻辑散落在镜像或 Compose 中。

常见示例：

- `WAIT_FOR_DB`
- `RUN_DB_MIGRATIONS`
- `DB_WAIT_TIMEOUT`
- `APP_INTERNAL_PORT`

要求：

- 项目应在部署文档中说明这些变量的作用
- 如果应用会自动迁移数据库，必须在文档中写明
- 如果服务 ready 依赖外部资源，也应明确写出等待策略

## 8. 标准命令约束

建议统一保留以下命令族：

### 8.1 顶层统一命令的语义

#### `make dev`

含义：

- 项目的默认开发入口
- 对开发者来说，应当是“进入标准开发态”的第一命令

要求：

- 全量 Docker 模式下，通常负责拉起完整开发栈
- 混合开发模式下，至少负责拉起容器侧依赖，并明确本地还需启动哪些进程
- 不能把真实开发入口分散到多个私有脚本里，却让 `make dev` 只做占位动作

#### `make build`

含义：

- 项目级默认构建入口

要求：

- 应输出该项目默认交付物
- 可以是镜像构建、后端编译产物、前端静态资源，或它们的统一编排入口
- 如果项目包含多个子应用，也必须保留一个项目级默认 `make build`

#### `make test`

含义：

- 项目级默认测试入口

要求：

- 应执行项目默认测试集合
- 可以进一步拆出 `make test-backend`、`make test-web` 等子命令
- 但不能只保留子命令而取消顶层 `make test`

#### `make deploy`

含义：

- 项目标准部署入口

要求：

- 应负责纳入统一发布链路的服务部署
- 如果部分服务不归它管理，必须在部署文档中说明真实发布方式
- 不能出现“命令名统一了，但实际没人知道它部署了什么”的情况

#### `make rollback`

含义：

- 项目标准回滚入口

要求：

- 应绑定明确的回滚单位
- 例如镜像标签、制品版本、Git 版本
- 文档中必须写清回滚前提、回滚命令和回滚后验证方式

#### `make logs`

含义：

- 项目默认日志查看入口

要求：

- 应优先面向主要应用服务
- 不应要求使用者先记住容器名或 Compose service 名
- 如果项目有多个关键服务，可以再提供更细的日志子命令

### 8.2 初始化类

- `make setup`
- `make init`

说明：

- `make setup`：初始化本地开发环境依赖
- `make init`：首次初始化入口，通常可以复用 `setup`

### 8.3 开发类

- `make dev`
- `make up`
- `make down`
- `make logs`

说明：

- `make up` / `make down`：允许单独启动或停止开发态容器
- `make logs`：允许单独查看默认应用日志
- 项目应提供 `make help` 作为命令发现入口

### 8.4 质量类

- `make test`
- `make build`

需要时可补充：

- `make lint`
- `make check`
- `make ci`

补充要求：

- `make build` 表示项目级默认构建入口
- `make test` 表示项目级默认测试入口
- 如果项目包含多个子应用，可以保留 `make build-admin`、`make build-store`、`make test-backend` 等子命令
- 但子命令不能替代顶层默认入口

### 8.5 部署类

- `make deploy-check`
- `make deploy`
- `make rollback`

说明：

- `make deploy-check`：执行部署前检查
- `make deploy`：执行标准部署
- `make rollback`：执行标准回滚

### 8.6 常见辅助子命令

项目可以按需补充以下辅助命令，方便开发者单独执行某个环节：

- `make help`
- `make up`
- `make down`
- `make deploy-check`
- `make prod-up`
- `make prod-down`
- `make prod-logs`
- `make db-up`
- `make db-down`
- `make db-shell`
- `make migrate`
- `make seed`
- `make lint`
- `make check`

要求：

- 这些命令用于暴露更细粒度操作
- 它们可以增强可发现性和可维护性
- 但不能替代统一顶层命令的职责

## 9. 项目允许扩展的边界

项目可以扩展：

- 多个前端服务
- 多个 Worker
- 反向代理
- 独立的构建 target
- 更复杂的部署脚本
- 额外的本地开发脚本
- 额外的部署自检脚本
- 分离式发布策略

但必须保留：

- 标准文件名
- 标准顶层命令
- 标准部署文档结构

## 10. 项目接入方式

新项目接入建议按下面顺序执行：

1. 复制模板目录
2. 替换项目名、服务名、镜像名、端口
3. 替换模板脚本中的占位启动命令
4. 填写 `.env.example` 和 `deploy/env/*.example`
5. 补写项目自己的 `deploy/README.md`
6. 验证 `make dev`、`make build`、`make test`、`make deploy`

旧项目接入建议按下面顺序执行：

1. 先统一命令命名
2. 再拆 Compose 分层
3. 再整理环境变量来源
4. 最后对齐部署文档

## 11. 禁止事项

以下做法不建议继续使用：

- 把复杂发布逻辑直接写进 `Makefile`
- 同时用一份真实 env 文件服务开发和生产
- 在 Compose 文件中写死密钥
- 每个项目自己发明一套顶层命令
- 只写操作说明，不提供可执行脚本
- 不记录项目使用的是哪一种本地运行模式
- 不记录哪些服务纳入标准部署、哪些服务独立发布

## 12. 交付要求

一套合格的项目基线，至少要满足：

- 开发、构建、测试、部署、回滚命令明确
- Compose 分层清晰
- 本地和部署变量分离
- 部署文档可执行
- 模板可复制
- 文档全部使用中文
- 本地运行模式有明确说明
- 服务发布拆分方式有明确说明
- 健康检查与启动契约有明确说明
