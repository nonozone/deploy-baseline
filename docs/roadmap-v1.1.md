# Deploy Baseline v1.1 Roadmap

## 1. 目标

`v1.1` 的目标不是继续横向增加概念，而是把 `deploy-baseline` 从“已经能用”推进到“更稳定、更可验证、更容易在真实项目中重复落地”。

这一阶段重点解决四类问题：

- 基线是否真的经得起不同项目场景反复验证
- skill 和模板是否有统一、可执行的最低验收面
- 默认部署路径是否足够清晰，避免开发者自己拼装
- 当前稳定支持范围是否表达清楚，避免过度承诺

## 2. 当前基础

截至 `v1.0` 阶段，仓库已经具备以下基础能力：

- 通用基线规范与部署 SOP
- `deploy-baseline-kit` 主 skill
- 可复制模板与 skill 内同步模板资产
- 统一命令约定
- 健康检查与部署就绪约定
- GitHub Actions `ci.yml` 与 `deploy.yml` 模板
- 回滚版本语义与最低验证面说明

`v1.1` 应建立在这些已有资产之上，而不是重新设计一套新体系。

## 3. 版本目标

`v1.1` 建议聚焦三件事：

1. 提升可信度：让基线和 skill 在多个真实项目形态下可重复验证
2. 提升可操作性：让“验证是否达标”变成标准命令，而不是文档要求
3. 提升边界清晰度：明确默认推荐路径、数据库成熟度和适用范围

## 4. P0：优先完成

### P0-1 样板项目验证矩阵

目标：

- 用最小样板场景验证 `deploy-baseline-kit` 和 `template/` 的输出质量

建议覆盖：

- 空目录新项目
- 轻量已有项目
- 重度已有部署项目
- 前后端分离项目
- monorepo 子项目入口
- 非 PostgreSQL 的一个对照样例

任务清单：

- [ ] 设计样板项目目录结构与最小资产集合
- [ ] 在仓库中约定样板项目存放方式或外部示例仓库策略
- [ ] 为每个样板定义预期识别结果
- [ ] 为每个样板定义预期改造结果
- [ ] 为每个样板定义最低验证命令
- [ ] 记录哪些样板属于稳定通过，哪些仍为实验性

验收标准：

- 至少 4 个样板场景可重复执行
- 每个样板都有明确的输入、输出和验证结果记录
- PostgreSQL 路线至少覆盖新项目和已有项目两类场景

### P0-2 统一验证入口

目标：

- 提供一个面向基线本身的统一验证入口，例如 `make verify-baseline`

建议纳入：

- `bash -n`
- `docker compose config`
- `make help`
- 模板与 skill 资产同步检查
- GitHub Actions workflow 基本校验
- env 示例文件完整性检查

任务清单：

- [ ] 定义 `verify-baseline` 的职责边界
- [ ] 实现脚本或 `Makefile` 入口
- [ ] 约定缺少 Docker 或 Compose 时的降级行为
- [ ] 约定哪些检查必须失败退出，哪些只提示 warning
- [ ] 将该入口写入 `README.md` 和相关文档

验收标准：

- 本仓库可以通过一个统一命令执行最低验证面
- 失败信息能指出是脚本、Compose、模板同步还是 workflow 问题
- 文档与实际命令保持一致

### P0-3 PostgreSQL 稳定路线加固

目标：

- 把 PostgreSQL 路线打磨成最完整、最可信的默认产品线

重点补强：

- PostgreSQL 16/17/18 已知差异
- `PGDATA` 与 volume 挂载策略
- 升级与回滚注意事项
- README 与模板示例中的持久化说明

任务清单：

- [ ] 整理 PostgreSQL 大版本部署差异表
- [ ] 校正模板中的 PostgreSQL 默认约定
- [ ] 补充 `deploy/README.md` 中的存储与备份说明
- [ ] 补充 skill 参考文档中的 PostgreSQL 版本陷阱说明

验收标准：

- PostgreSQL 路线对外可明确标注为当前最稳定支持路径
- PostgreSQL 相关模板、skill、文档口径一致
- 对 major version 差异有明确说明，而不是隐含假设

## 5. P1：第二阶段增强

### P1-1 默认远程部署主路径收敛

目标：

- 给出一条默认推荐的生产部署主路径，降低选择成本

建议主路径：

1. GitHub Actions 构建镜像
2. 推送到镜像仓库
3. 远端主机拉取镜像并执行 `docker compose up -d`

任务清单：

- [ ] 明确默认推荐部署链路
- [ ] 补充远端环境前置要求说明
- [ ] 明确 secrets 最低集合
- [ ] 为 `deploy.yml` 和 SOP 统一表述
- [ ] 说明何时适合偏离默认链路

验收标准：

- 新项目可以沿一条清晰主路径走通部署
- CI/CD 文档、模板和 SOP 不出现互相冲突的描述

### P1-2 monorepo / 多服务项目规则

目标：

- 让 skill 在复杂目录结构下的根目录判断和改造边界更稳定

任务清单：

- [ ] 定义“仓库根”与“项目根”的判断规则
- [ ] 定义从子目录启动 skill 时的默认作用范围
- [ ] 定义共享 `deploy/` 与局部 `deploy/` 的处理策略
- [ ] 定义多服务项目中主服务与依赖服务的识别规则
- [ ] 将这些规则补进 skill 参考文档

验收标准：

- 对 monorepo 和多服务项目，skill 的识别结果更稳定
- 输出方案时能明确说明本次改造影响范围

### P1-3 宿主机集成模板

目标：

- 把反向代理和系统服务的“最后一公里”也纳入安全模板

建议覆盖：

- Caddy
- Nginx
- systemd service
- 备份、校验、reload 示例

任务清单：

- [ ] 提供 merge 风格的反向代理示例
- [ ] 提供 systemd 最小样板
- [ ] 增加“先备份、再校验、再 reload”的 SOP 示例
- [ ] 补充 skill 里的系统级配置保护说明

验收标准：

- 用户可直接参考宿主机侧集成示例，而不是自行摸索
- 文档和 skill 都明确禁止无确认整文件覆盖

## 6. P2：产品化完善

### P2-1 数据库成熟度分级

目标：

- 对不同数据库路线采用统一的成熟度表达，而不是笼统写“支持”

建议分级：

- PostgreSQL：稳定支持
- MySQL / MariaDB：基础支持，待更多验证
- MongoDB / 外部托管数据库：实验性或按需求驱动

任务清单：

- [ ] 在 README 中明确成熟度分级
- [ ] 在 `docs/deploy-baseline-kit.md` 中统一表达
- [ ] 在 skill 文案中同步成熟度口径

验收标准：

- 仓库内所有对数据库支持范围的表述一致
- 不再出现容易被误解为“全面稳定支持”的表述

### P2-2 案例化文档

目标：

- 让读者快速理解这个仓库在真实项目里到底解决了什么问题

任务清单：

- [ ] 输出 2 到 3 个真实或半真实案例
- [ ] 每个案例包含改造前、识别结果、改造方案、最终产物
- [ ] 说明哪些步骤由 skill 自动完成，哪些仍需开发者提供信息

验收标准：

- 新用户读完案例后，能快速理解适用场景与预期结果

### P2-3 skill 输出契约标准化

目标：

- 让 `deploy-baseline-kit` 的输出更稳定、更像产品而不是一次性对话

建议固定输出项：

- root path
- project classification
- deployment mode
- database type
- current assets
- missing baseline pieces
- recommended path
- highest-risk edits
- verification plan
- next command

任务清单：

- [ ] 在 `SKILL.md` 中补充固定输出要求
- [ ] 在行为边界文档中说明最终输出格式
- [ ] 用真实项目再校验输出稳定性

验收标准：

- 同类项目的 skill 输出结构保持稳定
- 用户更容易一次性完成确认

## 7. 非目标

`v1.1` 阶段不建议把以下内容作为主目标：

- 一次性覆盖所有数据库与中间件组合
- 设计过多可选部署后端
- 在没有真实项目验证前承诺复杂企业级场景全面支持
- 把 skill 拆成多个用户需要手动串联的子 skill

## 8. 推荐执行顺序

建议按下面顺序推进：

1. 样板项目验证矩阵
2. 统一验证入口
3. PostgreSQL 稳定路线加固
4. 默认远程部署主路径收敛
5. monorepo / 多服务项目规则
6. 宿主机集成模板
7. 数据库成熟度分级
8. 案例化文档
9. skill 输出契约标准化

## 9. 里程碑完成判定

当满足以下条件时，可认为 `v1.1` 达成：

- 至少一条稳定产品线可被重复验证
- 基线仓库有统一验证命令
- README、SOP、skill、模板对默认推荐路径表述一致
- 数据库支持范围与成熟度表达清楚
- 用户可以通过样板或案例快速理解适用方式

## 10. 下一步建议

如果按投入产出比排序，建议先从以下两项开始：

1. 样板项目验证矩阵
2. `make verify-baseline`

这两项会最快提升整个项目的可信度，也最能为后续 PostgreSQL 强化、monorepo 支持和宿主机集成提供稳定回归面。
