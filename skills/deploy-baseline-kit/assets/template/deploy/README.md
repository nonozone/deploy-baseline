# 项目部署规范

> 这个文件是项目自己的部署文档入口，编写时应遵守通用部署 SOP。

## 1. 适用范围

说明这个部署文档适用于哪个环境、哪些服务、哪些角色。

## 2. 前置条件

至少说明：

- 服务器要求
- Docker 与 Compose 版本要求
- 部署用户与目录要求
- 镜像仓库访问要求

## 3. 本地运行模式说明

必须说明当前项目采用哪一种方式：

- 全量 Docker
- 混合开发

如果是混合开发，还要补充：

- 哪些服务走 Docker
- 哪些服务本地运行
- `make dev` 会启动哪些内容
- 开发人员还需要额外执行哪些命令

## 4. 服务发布拆分说明

至少说明：

- 哪些服务归 `make deploy` 管理
- 哪些服务独立发布
- 前端是否独立发布
- 数据库是否和应用共用同一套 Compose

## 5. 目录与文件约定

至少说明：

- 项目部署目录
- 环境文件位置
- 数据目录或命名卷策略
- 日志查看方式

## 6. 环境变量说明

至少说明：

- 真实生产变量来源
- `deploy/env/app.prod.env` 中哪些变量必须替换
- 哪些变量属于敏感信息

## 7. 部署前检查

标准入口：

```bash
make deploy-check
```

如果项目包含数据库或存储，还应说明是否存在额外自检脚本，例如：

- 持久化目录检查
- 命名卷检查
- 运行中挂载检查

## 8. 标准发布流程

标准入口：

```bash
make deploy
```

需要明确写出项目是否还需要：

- 数据库迁移
- 静态资源发布
- 缓存预热
- 网关刷新

## 9. 启动契约与发布完成判定

至少说明：

- 是否等待数据库就绪
- 是否自动执行数据库迁移
- 启动契约变量有哪些
- 什么条件下可以认为发布完成

## 10. 日志与健康检查

建议至少写清：

- 查看哪个服务日志
- 使用什么命令
- 健康检查接口是什么
- 发布完成后如何确认服务可用

## 11. 回滚流程

标准入口：

```bash
make rollback
```

必须补充：

- 回滚单位
- 回滚命令
- 回滚后验证方式

## 12. 常见故障排查

至少覆盖：

- 环境文件缺失
- 端口冲突
- 容器启动失败
- 健康检查失败
- 数据库未就绪

## 13. 项目特有说明

把这个项目自己的特殊部署前置条件、依赖或限制写在这里。
